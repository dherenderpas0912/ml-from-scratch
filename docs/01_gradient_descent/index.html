01 / OVERVIEW

What does gradient descent do?

GD finds the model parameters (w, b) that minimize a loss function by repeatedly moving in the direction of steepest descent — opposite to the gradient.

w  ←  w  −  lr  ·  ∂L/∂w
b  ←  b  −  lr  ·  ∂L/∂b

At every step: which direction does the loss increase? Move the opposite way. Repeat until loss stops decreasing.

02 / THE MATH

How the gradient is calculated

For a linear model ŷ = Xw + b with MSE loss, here is the full derivation step by step.

1

Define the loss function (MSE)

L = (1/n) · Σ (ŷᵢ − yᵢ)²

We want to minimize L over all training samples. ŷ is the prediction, y is the true label.

2

Substitute the model ŷ = Xw + b

L = (1/n) · Σ (Xw + b − y)²

# define error vector for convenience
err = Xw + b − y     # shape: (n_samples,)
3

Differentiate w.r.t. w — chain rule

∂L/∂w = (2/n) · Xᵀ @ err

# Why Xᵀ ?
# Each weight wⱼ affects loss through feature column j.
# Xᵀ[j] = all values of feature j across n samples.
# Dot with err = how much feature j contributed to error,
# summed across all samples.

# Shape check:
# Xᵀ → (n_features, n_samples)
# err → (n_samples,)
# Xᵀ @ err → (n_features,) ✓ one gradient per weight
4

Differentiate w.r.t. b — simpler

∂L/∂b = (2/n) · Σ err

# b is a scalar added to every prediction.
# Its derivative is just the mean error across all samples.
5

Final Python implementation

def gradients(X, w, b, y):
  n = len(y)
  err = X @ w + b − y           # (n,)
  dw = (2/n) * (X.T @ err)     # (n_features,)
  db = (2/n) * np.sum(err)     # scalar
  return dw, db

def vanilla_gd(X, y, lr, epochs):
  w, b = np.zeros(X.shape[1]), 0.0
  for epoch in range(epochs):
    dw, db = gradients(X, w, b, y)
    w -= lr * dw
    b -= lr * db
  return w, b

04 / LEARNING RATE

What happens with different learning rates?

The learning rate controls step size. Too small = slow. Too large = overshoots and diverges.

lr = 0.01 — too small

Converges correctly but takes many epochs. Gradient is accurate but steps are tiny.

lr = 0.15 — just right

Smooth, fast convergence. Loss drops steadily to near zero in few epochs.

lr = 0.95 — too high

Overshoots the minimum. Loss oscillates or diverges instead of converging.

05 / VARIANTS

Batch GD vs SGD vs Mini-batch GD

All three variants minimize the same loss function using the same update rule. The only difference is how many training samples are used to compute the gradient at each update step. This single choice has major consequences for speed, memory, stability, and convergence behaviour.

Batch GD — vanilla

Batch GD computes the gradient using the entire training dataset at every single update step. Because the gradient is averaged over all n samples, it is mathematically exact — it points precisely in the direction of steepest descent on the true loss surface.

The upside is smooth, stable convergence. The loss curve decreases monotonically (assuming a reasonable learning rate), and for convex loss functions like MSE, it is guaranteed to find the global minimum. There is no noise or randomness in the update direction.

The downside becomes clear at scale. If you have 1 million training samples, you must compute predictions and errors for all 1 million rows before taking a single weight update. This is computationally expensive and slow. For large datasets, one epoch can take minutes, and the model learns nothing until that epoch completes.

for epoch in range(epochs):
  err = X_all @ w + b − y_all    # full dataset — all n samples
  dw = (2/n) * X_all.T @ err   # exact gradient — no noise
  w -= lr * dw                 # one update per epoch
Stochastic GD — SGD

SGD goes to the opposite extreme — it computes the gradient using just one randomly chosen sample per update. Instead of waiting to see the whole dataset, the model updates its weights after every single data point. In one epoch of 10,000 samples, the weights are updated 10,000 times, compared to just once with Batch GD.

This speed is SGD's biggest strength. It starts learning immediately and can handle datasets that don't fit in memory, since it only needs one sample at a time. However, because the gradient is estimated from a single sample, it is very noisy — it may point in approximately the right direction but rarely the exact right one. The loss curve looks jittery and oscillates up and down even as it trends downward overall.

Surprisingly, this noise is sometimes useful. In non-convex loss surfaces (like neural networks), the random fluctuations can help the optimizer escape shallow local minima or saddle points that Batch GD would get stuck in. However, SGD may never fully settle — it keeps bouncing around near the minimum rather than converging cleanly onto it.

for epoch in range(epochs):
  for i in np.random.permutation(n):  # shuffle samples each epoch
    xi, yi = X[i:i+1], y[i:i+1]     # pick 1 sample
    err = xi @ w + b − yi         # error on that sample only
    dw = (2/1) * xi.T @ err      # noisy gradient estimate
    w -= lr * dw                 # n updates per epoch
INDUSTRY STANDARD

Mini-batch GD

Mini-batch GD is the practical middle ground that nearly all modern machine learning uses in production. Instead of using all samples or just one, it groups the data into small batches of 32 to 256 samples and computes the gradient over each batch. The batch size is a hyperparameter you tune — common defaults are 32 or 64.

With a batch of 32 samples, the gradient is a much better estimate than SGD's single-sample guess, but you still update the weights many times per epoch instead of just once. On a dataset of 10,000 samples with batch size 32, you get ~312 weight updates per epoch. This frequency of updates means the model learns much faster than Batch GD.

Critically, Mini-batch GD is designed to take advantage of modern GPU hardware. GPUs are built to process matrix operations in parallel, and a batch of 32–256 samples can be processed simultaneously in a single matrix multiply — far more efficiently than looping sample-by-sample as SGD does. This is why deep learning frameworks like PyTorch and TensorFlow use mini-batch GD by default. The loss curve is smooth enough to track progress clearly, fast enough to converge quickly, and noisy enough to avoid the worst local minima traps.

for epoch in range(epochs):
  for batch in get_batches(X, batch_size=32):  # ~n/32 updates per epoch
    err = batch @ w + b − y_batch   # error on 32 samples
    dw = (2/32) * batch.T @ err    # stable approx gradient
    w -= lr * dw                   # GPU-friendly matrix op
